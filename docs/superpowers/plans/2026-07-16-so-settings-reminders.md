# Раздел «Настройки» поставщика + управление напоминаниями: Implementation Plan

> REQUIRED SUB-SKILL: superpowers:subagent-driven-development. Шаги — чекбоксы.

**Goal:**
1. Собрать разрозненные настройки поставщика в отдельную вкладку «⚙️ Настройки» (реорганизация UI, без изменения поведения).
2. Добавить управление напоминаниями ПО ПОСТАВЩИКУ: какие напоминания слать (вечернее / 3ч / 2ч / 1ч / 30м / истёк) и по каким каналам (Telegram, пуш). Крон уважает эти настройки; при отсутствии настроек — поведение ровно как сейчас (все напоминания, только Telegram).

**Контекст (проверено в коде):**
- Вкладки менеджера поставщика в `src/views/SupplierOrdersManagerView.vue`: overview / status / list / schedules / templates (кнопки ~10–22, тела `pageTab==='...'`). Отдельной «settings» нет.
- Настройки СЕЙЧАС разбросаны:
  - Вкладка «Статус», верхняя панель (~111–135): приём вкл/пауза (`settings.is_accepting_orders`, кнопка `toggleAccepting`), текст паузы, чекбоксы «Авто-подача предыдущей» (`toggleAutoSubmit`, `auto_submit_previous`) и «Авто-письмо в дедлайн» (`toggleAutoEmail`, `auto_email_summary`). Разовое продление дедлайна и список переопределений (~154–175) — ОПЕРАТИВНОЕ, ОСТАЁТСЯ на «Статусе».
  - Вкладка «Графики» (~398–423): «Получатели итоговой сводки» (`notifyUsers`, `allNotifyUsers`) — ПЕРЕЕЗЖАЕТ в «Настройки». Дедлайны по дням недели, сетка графика, временный график — ОСТАЮТСЯ на «Графиках».
- Бэкенд: таблица `so_supplier_settings` (is_accepting_orders, auto_submit_previous, auto_email_summary, default_deadline_time, pause_message). Функции `soGetSupplierSettings` (~59), `soGetSupplierNotifyUsers`, `soSaveSupplierNotifyUsers`. Эндпоинты `settings` GET (~1737, отдаёт settings+overrides+notify_users) и `settings` POST (~1761, сохраняет settings + notify_users). Почта поставщика — `suppliers.email`/`cc_emails` (правится при подключении; отдельного редактора нет — в «Настройках» показать справочно, read-only, БЕЗ нового эндпоинта).
- Напоминания в кроне `api/cron_telegram.php`, блок «ЗАЯВКИ ПОСТАВЩИКАМ … напоминания» (~830–1072). Тайминги (~992–1008): `evening` (18:00 накануне), `expired` (истёк, -0.1..-10 мин), `3h`/`2h`/`1h`/`30m` (окна вокруг дедлайна). Рассылка ~1046–1067: собранным `$chatIds` ресторана (из `ro_telegram_subs` с `notify_so_reminders=1`), через `tgSend($chatId, $msgText, true, $keyboard)` с auth-токеном и кнопками. Дедуп через `tg_notification_log` (~1012–1015). ПУШ здесь сейчас НЕ шлётся.
- Пуш ресторану уже есть: `pushSendToRestaurant($pdo, $rNum, $rGroup, $payload)` — используется кнопкой «🔔 Напомнить» (`so/admin/remind-unsubmitted`) и подтверждением подачи. Проверить точную сигнатуру и то, доступна ли функция в контексте `cron_telegram.php` (подключить нужный include при необходимости).

**Architecture:**
- Хранилище настроек напоминаний — 2 новых столбца в `so_supplier_settings` (миграция, применяет Claude):
  - `reminder_offsets` VARCHAR(120) NULL — CSV включённых таймингов из белого списка `evening,3h,2h,1h,30m,expired`. NULL/пусто = ВСЕ включены (текущее поведение).
  - `reminder_channels` VARCHAR(40) NULL — CSV включённых каналов из белого списка `tg,push`. NULL/пусто = `tg` (текущее поведение).
- `soGetSupplierSettings` возвращает распарсенные `reminder_offsets` (массив) и `reminder_channels` (массив) с применением дефолтов. `settings` POST их валидирует (по белым спискам) и сохраняет.
- UI: новая вкладка «⚙️ Настройки». Переносим туда существующие контролы приёма/паузы/авто-подачи/авто-письма и «Получателей сводки»; добавляем блок «Напоминания» (чекбоксы таймингов + чекбоксы каналов). Всё грузится/сохраняется через уже существующие `settings` GET/POST (расширенные).
- Крон: для каждого поставщика в блоке напоминаний читает `reminder_offsets`/`reminder_channels` (одна выборка на поставщика, кэш в массиве); пропускает выключенные тайминги; шлёт по включённым каналам — TG (как сейчас) и/или пуш (`pushSendToRestaurant`). Email НЕ трогаем (решение пользователя).

## Global Constraints
- **Регрессия — критерий №1:** поставщик БЕЗ новых настроек (NULL столбцы) получает ровно текущее поведение — все тайминги, только Telegram, тот же текст/кнопки/дедуп. Реорганизация UI не меняет что и как сохраняется (те же поля `so_supplier_settings`, тот же список получателей).
- Русский UI, понятные подписи (пользователь не программист).
- Бэкенд PHP и крон живые из `api/`; фронт — `npm run build`. Миграцию применяет Claude вручную. Во время разработки реальные напоминания ресторанам не провоцировать.
- Белые списки таймингов/каналов — строго фиксированные; неизвестные значения отбрасывать (защита от мусора и инъекций; всё параметризовано).
- Дедуп напоминаний сохранить: ключ `so_rem_{type}_{sup}_{rest}_{date}` уже общий на все каналы — повторно не слать в течение 24ч. При добавлении пуша убедиться, что дедуп не даёт дублей и не блокирует «другой канал того же типа» некорректно (обсудить в задаче: логично дедупить по (type,sup,rest,date) единожды на весь тик, канал — деталь доставки).
- Не сломать: подачу заявок, авто-подачу предыдущей, авто-письмо, отправку сводок, графики/дедлайны.
- Пуш не имеет per-type флага отписки (как отмечено в проекте) — это ок, канал включается настройкой поставщика.

## File Structure

- `api/cron_telegram.php` — блок so-напоминаний: чтение настроек поставщика, фильтр таймингов, отправка по каналам (TG + пуш).
- `src/views/SupplierOrdersManagerView.vue` — новая вкладка «Настройки»: перенос контролов + блок «Напоминания»; удаление «Получателей сводки» с «Графиков»; перенос приёма/паузы/авто-* со «Статуса».
- `src/stores/supplierOrderStore.js` — при необходимости расширить методы settings (передача reminder_*).

## Task 1: Миграция + бэкенд настроек напоминаний

**Files:** Create `migrations/2026-07-16_so_reminder_settings.sql`; modify `api/includes/supplier_orders.php`

- [ ] **Step 1 (миграция):** `ALTER TABLE so_supplier_settings ADD COLUMN reminder_offsets VARCHAR(120) NULL DEFAULT NULL, ADD COLUMN reminder_channels VARCHAR(40) NULL DEFAULT NULL;` с комментарием: NULL = дефолт (все тайминги / только TG). Идемпотентность по возможности (проверка существования — не критично, применяет Claude вручную).
- [ ] **Step 2 (чтение):** В `soGetSupplierSettings` добавить в SELECT `reminder_offsets, reminder_channels`; в выдачу — распарсенные массивы: `reminder_offsets` → массив из CSV, пересечённый с белым списком `['evening','3h','2h','1h','30m','expired']`; если столбец NULL/пусто → ВЕСЬ белый список (дефолт). `reminder_channels` → массив из CSV ∩ `['tg','push']`; NULL/пусто → `['tg']`. Дефолтная ветка функции (когда строки настроек нет вообще) — тоже отдаёт эти дефолты.
- [ ] **Step 3 (сохранение):** В `settings` POST принять `reminder_offsets` и `reminder_channels` (массивы или CSV из тела), провалидировать по белым спискам (отбросить неизвестное, сохранить порядок белого списка), сериализовать в CSV и записать в те же INSERT…ON DUPLICATE KEY UPDATE (добавить столбцы). Пустой массив таймингов допустим (= напоминания выключены полностью); пустой массив каналов трактовать как «нет рассылки» ИЛИ вернуть к дефолту — РЕШЕНИЕ: пустой список каналов = напоминания не шлются (осознанный выбор пользователя выключить все каналы). Существующие поля (is_accepting/auto_*/pause) сохранять как раньше.

**Verification:** `php -l` чисто; миграция применяется; GET settings возвращает массивы с корректными дефолтами при NULL; POST сохраняет только валидные значения; мусор/инъекции отброшены; регрессия — поставщик без настроек отдаёт полный список таймингов и `['tg']`.

## Task 2: Крон — напоминания по настройкам поставщика (тайминги + каналы)

**Files:** Modify `api/cron_telegram.php`

**Контекст:** блок so-напоминаний ~830–1072. `$supId`, `$restNum`, `$deliveryDate`, `$reminderType`, `$chatIds`, `$byRest[$restNum]['group']` доступны. Отправка TG — `tgSend($chatId, $msgText, true, $keyboard)`.

- [ ] **Step 1 (чтение настроек поставщика):** Один раз на поставщика (кэш в локальном массиве по `$supId`) прочитать `reminder_offsets`/`reminder_channels` из `so_supplier_settings` (с теми же дефолтами: NULL → все тайминги / `tg`). Не делать выборку на каждый ресторан.
- [ ] **Step 2 (фильтр таймингов):** После определения `$reminderType` — если его нет в списке включённых таймингов поставщика, `continue` (пропустить). Дефолт (нет настроек) → все включены → поведение как сейчас.
- [ ] **Step 3 (каналы):** Определить включённые каналы поставщика. Telegram: текущая рассылка через `tgSend` — выполнять ТОЛЬКО если `tg` в каналах. Пуш: если `push` в каналах — один раз на ресторан отправить `pushSendToRestaurant($pdo, (int)$restNum, $restGroup, $payload)` с коротким заголовком/текстом напоминания и, по возможности, тем же redirect. Проверить, что функция `pushSendToRestaurant` доступна в кроне (подключить include, если нужно); если недоступна без риска — вынести отправку так, чтобы сбой пуша не ломал TG и наоборот (try/catch вокруг каждого канала).
- [ ] **Step 4 (дедуп):** Оставить единый дедуп по ключу `so_rem_{type}_{sup}_{rest}_{date}` на весь тик напоминания (не плодить отдельные ключи на канал, иначе смысл 24ч-дедупа теряется). Убедиться, что при обоих каналах дедуп-запись пишется один раз после успешной попытки (как сейчас — логирование в `tg_notification_log`). Пуш-only случай (tg выключен) тоже должен уважать дедуп.

**Verification:** синтаксис PHP чист; при поставщике без настроек — тайминги и текст/кнопки TG идентичны прежним, пуша нет; при выключенном тайминге — этот тип не шлётся; при `push` в каналах — ресторан получает web-push; при выключенном `tg` — TG не уходит; сбой одного канала не мешает другому; дедуп не даёт повторов в 24ч.
