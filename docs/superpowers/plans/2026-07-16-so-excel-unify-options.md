# Объединение Excel-отчётов заявок поставщику + опции: Implementation Plan

> REQUIRED SUB-SKILL: superpowers:subagent-driven-development. Шаги — чекбоксы.

**Goal:** Свести два генератора Excel заявки поставщику (скачивание в браузере и отправка поставщику через node) к ОДНОМУ общему движку построения листа, чтобы поставщик получал такой же богатый файл, что и закупщик, и чтобы новые опции писались один раз. Добавить две необязательные опции (по умолчанию ВЫКЛ): «убрать пустые строки» и «паллеты/вес». Опции действуют на ОБА пути (скачивание и отправку).

**Контекст (проверено в коде):**
- Скачивание строится во фронте: `src/views/SupplierOrdersManagerView.vue`, функция `exportExcel` (~1857–2042). Богатый формат: несколько дат = несколько листов; объединённые товары (`p.is_grouped`, `p.source_skus` через `buildDisplayProducts`); подсветка ручных правок (`adminQtyStyle`, `v.isAdmin`), авто-заявок (`autoQtyStyle`, `isAutoSubmitted(row)`); пометки «Не подал»/«Не нужна» (`skipRest`); подытоги по городам + ИТОГО. Данные берёт из `products/restaurants/orderItems` (для выбранной даты) или `store.adminGetStatus(supplierId, date)` для остальных.
- Отправка поставщику строится на сервере: node-скрипт `scripts/build_so_order_xlsx.mjs` (единственный лист, БЕЗ объединения/авто-подсветки/«Не нужна»), которому JSON готовит PHP `soBuildSummaryXlsx` в `api/includes/supplier_orders.php` (~83–237). Вызовы отправки: `send-summary` (Telegram, ~3070) и `send-summary-email` (~3145, через `soSendSummaryEmail`).
- ОБА генератора используют одну библиотеку `xlsx-js-style`. Фронт грузит её ДИНАМИЧЕСКИ (`await import('xlsx-js-style')`), чтобы не раздувать основной бандл. Node импортирует статически.
- Атрибуты паллет/веса уже обогащаются в `soBuildSummaryXlsx` (commit 145a632): у `products_map`/`products` есть `qty_per_box`, `boxes_per_pallet`, `weight_netto`, `weight_brutto` (вес — на КОРОБКУ в ГРАММАХ). Для СКАЧИВАНИЯ этих атрибутов у `products` из `adminGetStatus` пока НЕТ — нужно добавить.
- Единица количества — ШТУКИ. Коробок = qty / qty_per_box; паллет (доля) = коробок / boxes_per_pallet; вес кг = коробок × weight_* / 1000. Все атрибуты в справочнике заполнены (100%), но у отдельных SKU без карточки могут быть 0 — тогда строку веса по товару оставлять пустой.
- Полный репозиторий присутствует на сервере (`/var/www/bk-calc` — git-репо), поэтому node-скрипт из `scripts/` может импортировать общий модуль из `src/lib/` по относительному пути. `xlsx-js-style` есть в node_modules на сервере (скрипт им уже пользуется).

**Architecture:**
- Новый ЧИСТЫЙ общий модуль `src/lib/soOrderXlsx.js`: функция `buildSoOrderSheet(XLSX, opts)`, строящая ОДИН лист (worksheet) из данных одной даты. `XLSX` передаётся аргументом (dependency injection) — модуль сам библиотеку НЕ импортирует, чтобы сохранить динамический импорт во фронте и работать в node. Внутри — вся текущая богатая логика фронта (стили, города, подытоги, ИТОГО, объединённые товары, подсветка admin/auto, пометки) + две опции.
- Фронт `exportExcel`: тонкий цикл по датам — для каждой готовит `{products, restaurants, items, options, ...}` и зовёт `buildSoOrderSheet`, добавляет лист; читает состояние двух галочек-опций.
- Node `build_so_order_xlsx.mjs`: читает JSON (теперь + `options`), зовёт тот же `buildSoOrderSheet`, пишет буфер. Богатый формат автоматически достаётся поставщику.
- PHP `soBuildSummaryXlsx`: в JSON-payload добавляет `options` (прокинутые из эндпоинтов) — данные и обогащение НЕ меняются структурно. Эндпоинты `send-summary` и `send-summary-email` принимают флаги опций из тела запроса (по умолчанию ВЫКЛ) и передают в `soBuildSummaryXlsx`/`soSendSummaryEmail`.
- UI: две галочки рядом с кнопкой скачивания (обе по умолчанию OFF). Их значения идут и в скачивание, и в вызовы отправки (`adminSendSummary`, `adminSendSummaryEmail`).

## Global Constraints
- **Регрессия — критерий №1:** при ОБЕИХ опциях ВЫКЛ скачиваемый файл должен получаться идентично текущему (та же вёрстка, стили, листы, объединения, подсветка). Это главный признак, что рефактор ничего не сломал.
- Русский UI и подписи. Пользователь не программист — в отчёте понятные подписи строк.
- Бэкенд PHP живой из `api/`; node-скрипт живой из `scripts/`; фронт — `npm run build` (подменяет `dist`, идёт в прод). Во время разработки поставщикам ничего НЕ отправлять.
- Справочник (веса/паллеты) фильтровать по ГРУППЕ юрлиц поставщика (`getEntitiesInGroup`), как в `soBuildSummaryXlsx`; НЕ по одному юрлицу. Рабочие заявки — как сейчас.
- Опции по умолчанию ВЫКЛ. «Паллеты/вес»: у товара без атрибутов (0) строку веса оставлять пустой, не делить на 0.
- Общий модуль `src/lib/soOrderXlsx.js` — чистый (без обращений к DOM/файловой системе/сети), библиотеку XLSX принимает аргументом.
- Не ломать текущие критичные пути отправки (Telegram/почта поставщику) и подачу заявок.

## File Structure
- `src/lib/soOrderXlsx.js` — НОВЫЙ общий модуль построения листа (стили + логика + 2 опции).
- `src/views/SupplierOrdersManagerView.vue` — `exportExcel` переводится на общий модуль; 2 галочки-опции; передача опций в скачивание и в вызовы отправки.
- `scripts/build_so_order_xlsx.mjs` — переводится на общий модуль; читает `options` из JSON.
- `api/includes/supplier_orders.php` — `soBuildSummaryXlsx` добавляет `options` в payload; эндпоинты `send-summary`/`send-summary-email` читают флаги; при необходимости — прокидывание через `soSendSummaryEmail`.
- `src/stores/supplierOrderStore.js` — `adminSendSummary`/`adminSendSummaryEmail` передают флаги опций.
- (adminGetStatus — добавить атрибуты паллет/веса в `products`, чтобы скачивание могло считать вес.)

## Task 1: Общий модуль построения листа + перевод скачивания на него

**Files:** Create `src/lib/soOrderXlsx.js`; modify `src/views/SupplierOrdersManagerView.vue`

**Цель:** вынести ВСЮ логику построения листа из `exportExcel` в `buildSoOrderSheet(XLSX, opts)` без изменения результата (при опциях ВЫКЛ). Опции пока прокинуть как параметры со значением по умолчанию «выкл» — сами галочки в UI появятся в Task 3, здесь достаточно поддержки в модуле.

- [ ] **Step 1 (модуль):** Создать `src/lib/soOrderXlsx.js` с `export function buildSoOrderSheet(XLSX, { supplierName, dateFmt, products, restaurants, items, isAutoSubmitted, options })` → возвращает `ws`. Перенести из `exportExcel` (~1872–2026): все определения стилей, группировку по городам, заголовок, строки данных, подытоги, ИТОГО, применение стилей, `!merges`, `!cols`, `!rows`. Вход:
  - `products`: массив отображаемых товаров (уже прошедших `buildDisplayProducts`) с полями `sku`, `product_name`, `is_grouped`, `source_skus`, и (опционально) `qty_per_box`, `boxes_per_pallet`, `weight_netto`, `weight_brutto`.
  - `restaurants`: массив с `number`, `city`/`region`, `address`, `order_status`.
  - `items`: lookup-функция `getQty(r, p) → { qty, isAdmin } | null` ИЛИ данные, из которых модуль её построит — на усмотрение реализации, но поведение `getQty`/`skipRest` сохранить точно как в оригинале.
  - `isAutoSubmitted`: функция `(row) → bool` (для `autoQtyStyle`). Во фронте передаётся существующая; в node — всегда `false`.
  - `options`: `{ dropEmptyRows = false, showPalletWeight = false }`.
- [ ] **Step 2 (опция dropEmptyRows):** когда `true` — исключать из листа строки ресторанов, у которых заявка есть, но по всем товарам количество 0/пусто (та же проверка, что `skipRest`), а также, по смыслу «убрать пустые», рестораны без заявки (не подавшие). Город без оставшихся строк — не выводить. Подытоги/ИТОГО считать по оставшимся. При `false` — поведение как сейчас (не подавшие показываются красным, «Не нужна» помечается).
- [ ] **Step 3 (опция showPalletWeight):** когда `true` — после строки ИТОГО добавить блок из 4 строк по каждому товару (значение в его колонке), подписи в колонке №/Адрес: «Коробок» (сумма штук по товару / qty_per_box), «Паллет (доля)» (коробок / boxes_per_pallet), «Вес нетто, кг» (коробок × weight_netto/1000), «Вес брутто, кг» (коробок × weight_brutto/1000). Округление: коробки — до 2 знаков, доля паллеты — до 2 знаков, кг — до 1 знака; хвостовые нули убирать. У товара без атрибутов (0) соответствующие ячейки оставлять пустыми (не делить на 0). Оформить строки в стиле подытогов/ИТОГО (аккуратно, читаемо). При `false` — блока нет.
- [ ] **Step 4 (перевод фронта):** в `exportExcel` заменить тело цикла по датам на подготовку входа и вызов `buildSoOrderSheet(XLSX, {...})`; передать `isAutoSubmitted` и `options` (пока из констант `{dropEmptyRows:false, showPalletWeight:false}` — реальные галочки в Task 3). Многолистовость, имена листов, имя файла, тосты — сохранить. Динамический `await import('xlsx-js-style')` сохранить, `XLSX` передавать в модуль.
- [ ] **Step 5:** `npm run build`.

**Verification (после сборки):** при опциях ВЫКЛ скачанный файл идентичен прежнему (несколько дат = листы, объединённые товары, подсветка admin/auto, «Не подал»/«Не нужна», подытоги/ИТОГО, ширины/высоты). При `dropEmptyRows` — пустые/неподавшие рестораны и пустые города исчезают, итоги пересчитаны. При `showPalletWeight` — внизу 4 строки на товар с верными числами; товар без атрибутов даёт пустые ячейки; деления на 0 нет. `buildSoOrderSheet` — чистая (XLSX аргументом, без DOM/fs).
