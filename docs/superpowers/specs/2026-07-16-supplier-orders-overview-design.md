# Этап 2 — Обзор по всем поставщикам (модуль «Заявки поставщикам» / `/supplier-orders`)

Дата: 2026-07-16
Статус: утверждён к реализации

## Контекст

Панель закупщика (`SupplierOrdersManagerView.vue`) сейчас работает по одному
поставщику: закупщик выбирает поставщика в выпадающем списке, затем смотрит
вкладки «Статус / Список заявок / Графики / Шаблоны». Общей картины «что
происходит у всех поставщиков за день» нет — чтобы понять, кто уже подал заявки,
у кого скоро дедлайн и где день закрыт, приходится перебирать поставщиков по одному.

Это второй из четырёх этапов апгрейда модуля (Этап 1 — почта поставщику — закрыт).

## Цель этапа

Дать закупщику **единый экран-обзор**: строка = поставщик, за выбранный день видны
дедлайн с отсчётом, сколько ресторанов подало из скольких ожидается, и быстрые
действия (отправить сводку на Email/Telegram, продлить/закрыть день) — без захода
в карточку каждого поставщика.

## Решения (согласованы с пользователем)

- **Что показывать по строке:** дедлайн + обратный отсчёт + метка открыт/закрыт;
  «подано X из Y» с цветовой подсветкой; кнопки отправки сводки (Email и Telegram);
  продлить/закрыть день.
- **Период:** только один выбранный день (дата выбирается явно, по умолчанию сегодня).
  НЕ неделя-сеткой.
- **Напоминания ресторанам** («напомнить не подавшим») — НЕ в этом этапе, перенесены
  на Этап 3.
- **Размещение:** новая вкладка **«Обзор»**, первой в списке вкладок, открывается по
  умолчанию при входе в панель.

## Что переиспользуется (не пишем заново)

- `soCalculateDeadline($pdo, $supplierId, $deliveryDate)` (`api/includes/so_deadline.php`)
  — возвращает `is_closed`, `forced_closed`, `deadline_dt` (DateTime с таймзоной),
  `deadline_str`, `deadline_time`, `status` ('open'|'closed'). Источник дедлайна и
  статуса дня.
- Список поставщиков группы с фильтром доступа —
  `soAppendAllowedSupplierGroupFilter($sessionUser, $legalEntity, $where, $params)`
  (как в эндпоинте `so/admin/suppliers`).
- Подсчёт ожидаемых/подавших ресторанов — логика уже есть в обработчике
  `so/admin/status` и в `soBuildSummaryXlsx()`: ожидаемые рестораны берутся из
  эффективного графика (`soGetEffectiveScheduleRows`), поданные — из `so_orders` по
  `supplier_id + delivery_date + legal_entity IN (группа)`.
- RBAC модуля: блок `if ($soAction === 'admin')` уже проверяет доступ к
  `supplier-orders` (GET → view, POST → edit) до маршрутизации `$adminAction`.
- Фронтовые методы действий уже существуют в `supplierOrderStore.js` и принимают
  `supplierId` явно: `adminSendSummary(id, date)` (Telegram),
  `adminSendSummaryEmail(id, date)` (Email), `adminExtendDeadline(...)`,
  `adminCloseDay(id, date, isClosed)`.

## Функциональные требования

### 1. Бэкенд — эндпоинт обзора
- Новый маршрут `GET so/admin/overview?date=YYYY-MM-DD[&legal_entity=...]` внутри блока
  `$soAction === 'admin'` (RBAC уже применён, GET → уровень view).
- Берёт поставщиков той же выборкой, что `so/admin/suppliers` (`is_active=1`,
  `so_enabled=1`, фильтр доступа по группе). Для каждого поставщика за указанную дату
  считает лёгкий статус (без генерации Excel):
  - дедлайн: `deadline_str`, `deadline_time`, `is_closed`, `forced_closed`,
    `deadline_at` (ISO-8601 с offset из `deadline_dt`, либо `null` если день закрыт
    принудительно/нет правила) — фронту для отсчёта;
  - `expected_count` — сколько ресторанов ожидается по графику на этот день;
  - `submitted_count` — сколько из них подали заявку;
  - `is_accepting` — принимает ли поставщик заявки (пауза/нет);
  - `has_schedule` — есть ли вообще график на день (если нет — строка «нет поставки»).
- Возвращает `{ date, suppliers: [ { id, short_name, ... поля выше } ] }`,
  отсортированные по `short_name`.
- Дата по умолчанию (если параметр пуст/некорректен) — сегодня в таймзоне проекта.

### 2. Стор
- Метод `adminGetOverview(date, legalEntity = null)` в `supplierOrderStore.js` —
  GET на новый эндпоинт, возвращает массив `suppliers` (+ дату).

### 3. Фронтенд — вкладка «Обзор»
- Новая вкладка `pageTab === 'overview'`, **первой** в ряду вкладок, **по умолчанию**
  (`pageTab` инициализируется как `'overview'`; при заходе с конкретным supplierId в URL
  — открывать привычную «Статус», чтобы не ломать существующие ссылки).
- Наверху — выбор даты (`<input type="date">`, по умолчанию сегодня). Смена даты
  перезагружает обзор.
- Таблица, строка = поставщик:
  - **Поставщик** — клик проваливает в его вкладку «Статус» (ставит `currentSupplierId`,
    `pageTab='status'`, грузит статус).
  - **Дедлайн** — время + обратный отсчёт (тикает на клиенте из `deadline_at`) + метка
    «открыт»/«закрыт»/«день закрыт». Просроченный/закрытый — приглушённым/красным.
  - **Подано** — «X из Y», цвет: зелёный если X≥Y (и Y>0), жёлтый если 0<X<Y, красный
    если X=0 при Y>0; «— нет поставки» если `has_schedule=false`.
  - **Действия** — кнопки: ✉️ Email, ✈️ Telegram (обе шлют сводку за выбранный день
    этому поставщику), ⏰ Продлить, 🔒 Закрыть/Открыть день. У каждой — состояние
    «в процессе» и тост об успехе/ошибке.
- Кнопка «Обновить» перечитывает обзор.

### 4. Действия строки
- Email / Telegram — вызывают `adminSendSummaryEmail(id, date)` /
  `adminSendSummary(id, date)`. Email-кнопка неактивна, если у поставщика нет почты
  (как на вкладке «Статус»): для этого в ответе обзора отдаём `has_email` (bool).
- Продлить / Закрыть — переиспользуют существующую логику (модалка продления /
  подтверждение закрытия), но для конкретного поставщика строки; после действия —
  перезагрузка обзора.

## Данные (БД)

Миграций **не требуется** — все нужные данные уже есть (графики, `so_orders`,
`so_supplier_settings`, `suppliers`, дедлайн-правила/оверрайды).

## Крайние случаи и надёжность

- Таймзона: `deadline_at` формируется из `deadline_dt` (в нём уже таймзона проекта) в
  ISO-8601 с offset — фронт делает `new Date(deadline_at)` без ручной коррекции.
  Известная гоча PHP(UTC)/MySQL(+03:00) — не считать отсчёт на стороне PHP «сейчас минус»,
  отдавать целевой момент, отсчёт крутит фронт.
- Нет графика на день → `has_schedule=false`, `expected_count=0`; строка показывает
  «нет поставки», действия отправки сводки неактивны/бессмысленны (кнопки можно оставить,
  но сервер вернёт `empty`/`no_schedule` — тост это отразит).
- День принудительно закрыт → метка «день закрыт», кнопка становится «Открыть».
- Поставщик на паузе (`is_accepting=false`) → пометка в строке.
- Много поставщиков в группе — обзор считает статус в цикле по поставщикам; количество
  поставщиков невелико (единицы-десятки), допускается. Тяжёлую генерацию Excel в обзоре
  НЕ делаем.
- Права: эндпоинт внутри admin-блока, RBAC уже проверен; отдельно ничего не добавляем.
- Группа юрлиц: выборка поставщиков и подсчёт поданных — строго по группе поставщика
  (`getEntitiesInGroup`), как во всей панели.

## Проверка (ручная — автотестов в проекте нет)

1. Открыть панель закупщика → сразу видна вкладка «Обзор» со всеми поставщиками за сегодня.
2. Сменить дату → пересчёт «подано X из Y» и дедлайнов.
3. Отсчёт до дедлайна тикает; после дедлайна — метка «закрыт».
4. Клик по поставщику → проваливается в его «Статус».
5. Кнопки Email/Telegram из строки шлют сводку нужному поставщику (проверить тост и,
   для Email, запись в `so_email_log`).
6. Продлить/Закрыть день из строки → обзор обновляется, метка меняется.
7. Поставщик без почты → Email-кнопка неактивна.
8. Поставщик без графика на день → «нет поставки».
