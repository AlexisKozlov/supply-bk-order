# Этап 3 — Напоминания и сигналы ресторанам (модуль «Заявки поставщикам» / `/supplier-orders`)

Дата: 2026-07-16
Статус: утверждён к реализации

## Контекст

Продолжение апгрейда модуля. Этап 1 (почта поставщику) и Этап 2 (обзор по всем
поставщикам) закрыты. Сейчас закупщик видит, кто не подал заявку, но «подтолкнуть»
ресторан можно только вручную вне системы. Ресторан, подав заявку, не получает
подтверждения. Никаких авто-сигналов о приближении дедлайна ресторанам нет.

## Цель этапа (согласовано с пользователем)

Три функции, все — уведомления **ресторанам** по каналам **Telegram + Push** (без email):

1. **Напомнить не подавшим** — кнопка у закупщика: в один клик уведомить рестораны,
   которые ещё не подали заявку этому поставщику на выбранный день.
2. **Подтверждение подачи** — когда ресторан подаёт заявку, ему приходит подтверждение
   «заявка принята».
3. **Авто-сигналы ресторанам** — крон: «скоро дедлайн, подайте заявку» не подавшим и
   «приём закрыт» после дедлайна.

## Что переиспользуется (готовая инфраструктура)

- **Push ресторану:** `pushSendToRestaurant($pdo, $restaurantNumber, $legalEntityGroup, ['title','body','url','tag'])` (`api/includes/push_send.php`).
- **Telegram ресторану:** подписчики в `ro_telegram_subs` (поля `restaurant_number`,
  `legal_entity_group`, `chat_id`, `verified_at`, `must_reverify_by`), отправка через
  `tgClientSend(...)` / `sendTelegramMessage($botToken, $chatId, $html, 'HTML')`. Токен —
  `$_ENV['TELEGRAM_BOT_TOKEN']`. Готовый образец «TG+Push ресторану» — блок временного
  графика в `supplier_orders.php` (~строки 532–566).
- **Настройки уведомлений ресторана** (`ro_telegram_subs`, тумблеры в боте):
  - `notify_so_reminders` — «Напоминания о заявках» → для функций 1 и 3.
  - `notify_confirmations` — «Подтверждения заявок» → для функции 2.
  Уведомления ДОЛЖНЫ уважать эти флаги — ресторан может отписаться.
- **Расчёт ожидается/подано:** `soGetEffectiveScheduleRows` + запрос поданных из `so_orders`
  (как в `soBuildSummaryXlsx`/`soGetDayStatusLight`). Дедлайн — `soCalculateDeadline`.
- **Крон дедлайнов:** `api/cron_telegram.php` уже проходит по поставщикам у дедлайна
  (авто-подстановка + авто-письмо Этапа 1), токен `$BOT_TOKEN` уже загружен. Защита
  авто-дублей — таблица-сторож с INSERT IGNORE (как `so_auto_submit_log`, `so_email_auto_log`).

## Функциональные требования

### 1. Напомнить не подавшим (кнопка закупщика)
- Эндпоинт `POST so/admin/remind-unsubmitted { supplier_id, delivery_date }` (в admin-блоке,
  RBAC edit для POST уже проверен). Считает не подавших (ожидаемые минус подавшие) и каждому
  шлёт TG+Push с уважением `notify_so_reminders`. Возвращает `{ reminded, total_unsubmitted }`.
- Текст: «🔔 <Поставщик>: не забудьте подать заявку на <ДД.ММ>. Дедлайн <ЧЧ:ММ>.» Ссылка/кнопка
  в кабинет заявок ресторана. Push — короткий вариант того же.
- Только если приём открыт (день не закрыт, дедлайн не прошёл) — иначе тост «приём уже закрыт»
  (кнопку можно не дизейблить, сервер вернёт понятный ответ).
- Фронт: кнопка «🔔 Напомнить» в строке обзора (вкладка «Обзор») — активна, когда
  `submitted_count < expected_count` и день открыт; и такая же кнопка на вкладке «Статус».
  Тост с числом отправленных.

### 2. Подтверждение подачи (авто, при `so/submit-order`)
- После успешного `commit` в обработчике `so/submit-order` — уведомить подавший ресторан
  (TG+Push, уважая `notify_confirmations`).
- Текст: «✅ Заявка <Поставщик> на <ДД.ММ> принята.» Для `skip_delivery` — «✅ <Поставщик>,
  <ДД.ММ>: отмечено, что поставка не нужна.»
- Строго не блокирующее: обёрнуто в try/catch, ошибка уведомления НЕ роняет подачу заявки
  и не меняет ответ.

### 3. Авто-сигналы ресторанам (крон)
- В `cron_telegram.php`, по тем же поставщикам/дедлайнам, что и Этап 1:
  - **«Скоро дедлайн»**: если до дедлайна осталось ≤ LEAD минут (константа, старт 60) и приём
    открыт — не подавшим шлём «⏰ <Поставщик>: приём заявок на <ДД.ММ> закрывается в <ЧЧ:ММ>.
    Подайте заявку.» (уважая `notify_so_reminders`). Сторож `so_signal_log` тип `deadline_soon`.
  - **«Приём закрыт»**: когда дедлайн только что прошёл (окно 0..15 мин после, как авто-письмо)
    или день принудительно закрыт — не подавшим шлём «🔒 <Поставщик>: приём заявок на <ДД.ММ>
    закрыт.» (уважая `notify_so_reminders`). Сторож `so_signal_log` тип `closed`.
- Защита от повторов: `so_signal_log (supplier_id, delivery_date, signal_type)` UNIQUE + INSERT
  IGNORE перед рассылкой; если 0 строк — уже слали, выходим. Так каждый сигнал уходит один раз
  на (поставщик, день, тип).
- Порядок: сигналы «закрыт» — после блока авто-подстановки заявок (чтобы подставленные
  засчитались как поданные и им не улетело лишнее).

## Данные (БД)

Миграция `migrations/20260716_so_signal_log.sql`:
- `so_signal_log`:
  - `id` BIGINT PK AUTO_INCREMENT
  - `supplier_id` CHAR(36) NOT NULL
  - `delivery_date` DATE NOT NULL
  - `signal_type` VARCHAR(32) NOT NULL  (`deadline_soon` | `closed`)
  - `restaurants_notified` INT NOT NULL DEFAULT 0
  - `created_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP
  - `UNIQUE KEY (supplier_id, delivery_date, signal_type)`

Никаких изменений в `ro_telegram_subs` — флаги уже есть.

## Общие серверные детали

- **Хелпер рассылки** `soNotifyRestaurantOrders($pdo, $botToken, $rNum, $rGroup, $tgHtml, array $push, string $notifyCol)`:
  - TG: `SELECT DISTINCT chat_id FROM ro_telegram_subs WHERE restaurant_number=? AND legal_entity_group=? AND {$notifyCol}=1 AND (verified_at IS NOT NULL OR (must_reverify_by IS NOT NULL AND must_reverify_by > NOW()))` → `sendTelegramMessage`/`tgClientSend`. `$notifyCol` — из фиксированного белого списка (`notify_so_reminders`|`notify_confirmations`), НЕ из пользовательского ввода (защита от инъекции в имя колонки).
  - Push: `pushSendToRestaurant($pdo, $rNum, $rGroup, $push)`.
  - Возвращает число охваченных каналов/ресторанов; ошибки каналов не пробрасывает (лог + продолжаем).
- **Хелпер не подавших** `soGetUnsubmittedRestaurants($pdo, $supplierId, $supplierGroup, $deliveryDate): array` — ожидаемые рестораны на день (число+группа) минус подавшие (`so_orders.status != 'draft'`), как в `soGetDayStatusLight`. Возвращает список `[{restaurant_number, legal_entity_group}]`.

## Крайние случаи и надёжность

- Ресторан отписан от нужного флага → не получает (это by design).
- Нет TG-подписки и нет push → тихо пропускаем; в счётчик не идёт.
- Повтор крона → нет дублей сигналов (сторож `so_signal_log`).
- Ручное «Напомнить» можно жать повторно (без сторожа — закупщик сам решает); каждое нажатие
  шлёт заново.
- Подтверждение при повторной подаче той же заявки — уходит каждый раз (ресторан переотправил
  — получил подтверждение); допустимо.
- Группа юрлиц: рассылка строго по `legal_entity_group` ресторана (не слать чужой группе).
- Ошибка уведомления никогда не роняет подачу заявки/ответ эндпоинта.

## Проверка (ручная — автотестов нет)

1. Кнопка «🔔 Напомнить» в обзоре/статусе → не подавшие рестораны получают TG+Push;
   тост с числом; подавшие не получают.
2. Ресторан подаёт заявку → приходит подтверждение (при включённом `notify_confirmations`).
   При выключенном — не приходит. Подача не ломается, даже если бот недоступен.
3. Крон за LEAD минут до дедлайна → «скоро дедлайн» не подавшим, один раз.
4. Крон после дедлайна → «приём закрыт» не подавшим, один раз; повторный прогон — без дублей.
5. Отписанный от «Напоминания о заявках» ресторан не получает пункты 1 и 3.

## Вне объёма этапа

- Настраиваемое per-supplier время «скоро дедлайн» (пока константа LEAD=60 мин).
- Сигналы закупщикам (пользователь выбрал только рестораны).
- Email-подтверждения ресторану (выбран канал push/tg).
- Этап 4 — полировка списка заявок и скорость.
