# Онбординг разработчика

Этот документ нужен разработчику, который впервые открывает проект. Он подходит и для работы на текущем сервере, и для ситуации, когда репозиторий передали отдельно без боевой базы.

## Быстрое понимание проекта

`bk-calc` — портал закупок. Исторически он начинался как калькулятор заказа, но сейчас это большой рабочий портал:

- основной заказ поставщикам;
- кабинет ресторанов;
- заявки поставщикам;
- цены и ПСЦ;
- аналитика;
- графики доставки;
- тендеры;
- маркетинг;
- Telegram;
- загрузка машин;
- 1C Robot.

Фронтенд написан на Vue 3. Бэкенд — PHP без фреймворка. База — MariaDB/MySQL.

## Локальный запуск

Установить зависимости:

```bash
npm install
```

Запустить Vite:

```bash
npm run dev
```

Собрать production:

```bash
npm run build
```

Локальный dev-сервер поднимает только фронтенд. Для полноценной работы нужны PHP API и база.

## Что нужно для полноценной среды

Минимально:

- Node.js и npm;
- PHP с PDO MySQL;
- MariaDB/MySQL;
- веб-сервер для `/api`;
- файл `api/.env`;
- права записи в `api/uploads/`.

Дополнительно:

- Tesseract и PHP GD для OCR;
- cron для Telegram/ПСЦ/health-задач;
- Telegram Bot Token для Telegram-функций;
- Python-зависимости для `1c_robot_web`, если нужен этот подпроект.

## Секреты и конфиги

Файл `api/.env` не должен попадать в публичную документацию и коммиты. В нём могут быть:

- доступы к базе;
- CORS-настройки;
- Telegram-токены;
- другие секреты интеграций.

Для нового разработчика лучше дать пример `.env` без реальных паролей отдельно, если он нужен.

## Рабочий цикл изменения

1. Найти страницу или API-действие.
2. Проверить, какие stores и lib-файлы участвуют.
3. Проверить, есть ли серверная часть.
4. Проверить права и юрлица.
5. Если меняется база, добавить миграцию в `migrations/`.
6. Внести минимальные изменения.
7. Выполнить `npm run build`.
8. Ручно проверить сценарий.

## Как искать код

Полезные команды:

```bash
rg "имя_функции"
rg "db.rpc('имя_rpc'"
rg "db.from('имя_таблицы'"
rg "module: 'restaurant-orders'" src/router/index.js
rg "имя_таблицы" api src migrations
```

Если задача относится к странице, начинайте с `src/router/index.js`: там видно имя страницы, файл Vue-компонента и модуль прав.

## На что обратить внимание сразу

- `api/includes/rpc.php` очень большой. Изменения в нём нужно делать точечно.
- Логика юрлиц есть и на фронтенде, и на бэкенде.
- Права доступа тоже есть и на фронтенде, и на бэкенде.
- Автотестов нет.
- `dist/` не является исходником.
- `api/migrations/` — старый слой миграций, но он ещё копируется в сборку.

## Передача проекта без боевой базы

Если разработчик получает только репозиторий:

- фронтенд можно собрать и частично изучать без базы;
- большинство страниц не будет полноценно работать без API и данных;
- структуру таблиц нужно восстанавливать по `migrations/` и `api/migrations/`;
- реальные тестовые данные лучше готовить отдельно;
- секреты из `api/.env` заменяются на локальные.
