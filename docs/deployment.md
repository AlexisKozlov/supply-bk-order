# Запуск, сборка и деплой

## Локальный запуск фронтенда

```bash
npm run dev
```

Это запускает Vite dev-сервер. Фронтенд ожидает API по адресу `/api`.

## Production-сборка

```bash
npm run build
```

Команда из [package.json](/var/www/bk-calc/package.json):

```bash
rm -rf dist && vite build && mkdir -p dist/api && cp -r api/*.php api/includes api/migrations dist/api/
```

Что происходит:

1. удаляется старая папка `dist`;
2. собирается фронтенд;
3. создаётся `dist/api`;
4. PHP API копируется в сборку.

## Preview сборки

```bash
npm run preview
```

Это локальный просмотр production-сборки Vite. Для полноценной проверки API всё равно нужна серверная PHP/MariaDB-среда.

## Переменные окружения API

Файл:

```text
api/.env
```

Основные переменные:

```text
DB_HOST=...
DB_NAME=...
DB_USER=...
DB_PASS=...
CORS_ORIGIN=...
```

В `.env` также могут быть токены и настройки интеграций. Не публиковать реальные секреты в документации и коммитах.

## Требования к серверу

Нужны:

- PHP с PDO MySQL;
- MariaDB/MySQL;
- веб-сервер, который отдаёт фронтенд и прокидывает `/api` в PHP;
- права на запись в `api/uploads/`;
- для OCR — Tesseract и PHP GD;
- для cron-задач — cron или другой планировщик.

## Миграции

Миграции применяются вручную. Обычно новый SQL-файл добавляется в `migrations/`.

Перед применением:

- сделать резервную копию базы или убедиться, что она есть;
- проверить SQL на тестовой среде, если она доступна;
- проверить, что миграция не конфликтует с уже существующими полями;
- после применения проверить соответствующий модуль в интерфейсе.

## Ручная проверка после изменений

Минимум:

```bash
npm run build
```

Если менялся фронтенд:

- открыть нужную страницу;
- проверить вход пользователя;
- проверить права доступа;
- проверить мобильный или узкий экран, если менялась верстка.

Если менялся API:

- проверить успешный сценарий;
- проверить ошибку доступа;
- проверить пользователя без нужного юрлица;
- проверить, что старые сценарии рядом не сломались.

Если менялась база:

- применить миграцию;
- проверить чтение старых строк;
- проверить создание новых строк;
- проверить отчёты/выгрузки, которые используют эти поля.

## Что не править как исходники

Не вносить рабочие правки в:

- `dist/`;
- собранные файлы из `public/assets/`;
- временные backup-папки.

Исходники находятся в:

- `src/`;
- `api/`;
- `migrations/`;
- `scripts/`;
- `1C_Robot_Pro/`;
- `1c_robot_web/`.
