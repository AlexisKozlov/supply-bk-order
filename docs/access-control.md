# Права доступа

Права доступа распределены между фронтендом и бэкендом. При изменении ролей, видимости модулей или доступа к данным нужно проверять обе стороны.

## Где описаны права

Фронтенд:

- [src/stores/userStore.js](/var/www/bk-calc/src/stores/userStore.js)
- [src/router/index.js](/var/www/bk-calc/src/router/index.js)

Бэкенд:

- [api/includes/helpers.php](/var/www/bk-calc/api/includes/helpers.php)
- [api/includes/crud.php](/var/www/bk-calc/api/includes/crud.php)
- [api/includes/rpc.php](/var/www/bk-calc/api/includes/rpc.php)
- частично специализированные модули: `restaurant_orders.php`, `supplier_orders.php`, `truck_loading.php`.

## Уровни доступа

Используются уровни:

```text
full = полный доступ
edit = просмотр и изменение
view = только просмотр
none = нет доступа
```

Числовой порядок задаётся в `ACCESS_LEVELS`.

## Роли

Базовые роли:

- `admin`;
- `manager`;
- `user`;
- `viewer`.

Для ролей есть шаблоны прав по модулям. У конкретного пользователя могут быть индивидуальные переопределения в поле `permissions`.

## Проверка на фронтенде

Router проверяет:

- нужна ли авторизация;
- является ли пользователь админом для админских страниц;
- есть ли доступ к модулю страницы.

Если доступа нет, пользователь перенаправляется на первый доступный модуль или на главную.

Важно: фронтенд-ограничение нужно только для удобства интерфейса. Настоящая защита должна быть на сервере.

## Проверка на бэкенде

REST CRUD:

- определяет таблицу;
- находит связанный модуль;
- сравнивает уровень пользователя с нужным уровнем;
- проверяет доступ к юрлицу;
- запрещает запись в read-only таблицы.

RPC:

- для публичных действий авторизация не нужна;
- для приватных действий нужно проверять сессию;
- для важных операций используются `requireModuleAccess`, `checkLegalEntityAccess` и похожие проверки.

Специализированные модули сами проверяют доступ:

- ресторанные заказы;
- заявки поставщикам;
- загрузка машин;
- загрузки файлов.

## Юрлица и группы

У пользователя может быть ограничение по `legal_entities`. При таком ограничении он не должен видеть и менять данные чужого юрлица.

Особенно внимательно проверять:

- REST-запросы без фильтра `legal_entity`;
- создание и обновление записей;
- дочерние таблицы, где юрлицо надо получать через родителя;
- модули с `legal_entity_group`.

## Кабинет ресторанов

Кабинет ресторанов не использует обычную сотрудническую сессию. Там отдельная авторизация по `ro_token`, который передаётся как `X-RO-Token`.

Права ресторана ограничены его номером и группой юрлиц.

## Правило для изменений

Если меняется доступ к странице, таблице или действию:

1. проверить `src/router/index.js`;
2. проверить `src/stores/userStore.js`;
3. проверить `api/includes/helpers.php`;
4. проверить `api/includes/crud.php`;
5. проверить нужный RPC или специализированный PHP-модуль;
6. вручную проверить пользователя с недостаточными правами.
