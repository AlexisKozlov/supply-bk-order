# AGENTS.md

Этот файл задаёт постоянный рабочий контекст для Codex в этом репозитории.

## Язык

- Проект русскоязычный: UI, комментарии, данные и ответы пользователю вести на русском.

## Стиль общения

- Пользователь не программист.
- Все объяснения, описания изменений и сводки писать простым понятным языком.
- Избегать лишнего жаргона. Если без технического термина не обойтись, сразу пояснять его простыми словами.
- Писать коротко и по делу.

## Процесс работы

- Перед любыми изменениями сначала найти релевантные файлы и объяснить, где именно будет реализована задача.
- Затем предложить короткий пошаговый план и ожидаемую стратегию реализации.
- Не вносить изменения, пока этот план не показан пользователю.
- Если по задаче есть неясность, неоднозначность или риск сделать неверное допущение, сначала задать пользователю уточняющий вопрос.
- По важным моментам всегда задавать вопрос с готовыми вариантами ответа, чтобы пользователь мог выбрать один из вариантов.
- После этого можно переходить к правкам.

## Команды

```bash
npm run dev      # Vite dev-сервер фронтенда
npm run build    # Сборка фронтенда в dist/ и копирование PHP API в dist/api/
npm run preview  # Локальное превью production-сборки
```

## Что это за проект

- Это не только калькулятор заказа, а большой портал закупок.
- В проекте есть основной заказ, история, планирование, аналитика, база товаров, график доставки, реализация ресторанов, цены, тендеры, маркетинг, корректировки, чат, сбор остатков, дефицит, заказы ресторанов, заявки поставщикам, загрузка машин, Telegram и админка.

## Стек

- Фронтенд: Vue 3, Pinia, Vue Router, Vite, `vite-plugin-pwa`.
- Бэкенд: PHP без фреймворка, PDO, MariaDB/MySQL.
- Без TypeScript.
- Без CSS-препроцессоров и UI-фреймворков.

## Точки входа

- Фронтенд: `index.html` -> `src/main.js` -> `src/App.vue` -> `src/router/index.js`
- Бэкенд API: `api/index.php`

Важно:

- Корневого `index.php` в актуальной версии репозитория нет.
- Не опираться на старые описания, где он упоминается.

## Основные директории

- `src/views/` — страницы и крупные модули интерфейса.
- `src/stores/` — состояние приложения и клиентская бизнес-логика.
- `src/lib/` — расчёты, импорт/экспорт, API-клиент, общие утилиты.
- `src/components/` — переиспользуемые интерфейсные компоненты.
- `src/layouts/` — каркас приложения.
- `api/includes/` — серверная логика по модулям.
- `migrations/` — актуальные SQL-миграции и разовые скрипты миграции.
- `api/migrations/` — старый слой SQL-миграций, всё ещё копируется в сборку.
- `scripts/` — вспомогательные Node-скрипты, в основном генерация Excel.
- `sql/` — отдельные SQL-заготовки и исторические SQL-файлы.
- `public/` — статические файлы.
- `dist/` — результат сборки, не исходники.

## Ключевые модули

- `src/stores/orderStore.js` — основной заказ, позиции, undo/redo, загрузка и аудит.
- `src/lib/calculations.js` — формулы расчёта заказа и буферного режима.
- `src/lib/saveOrder.js` — сохранение и обновление заказа, diff, уведомления.
- `src/lib/apiClient.js` — REST/RPC-клиент фронтенда.
- `src/stores/userStore.js` — сессия, роли, права, скрытые модули, техработы.
- `api/includes/rpc.php` — большой RPC-модуль с основной серверной бизнес-логикой.
- `api/includes/crud.php` — REST CRUD для таблиц с проверками прав.
- `api/includes/helpers.php` — авторизация, RBAC, фильтры, общие серверные функции.
- `api/includes/restaurant_orders.php` — кабинет и заказы ресторанов.
- `api/includes/supplier_orders.php` — заявки поставщикам.
- `api/includes/truck_loading.php` — загрузка машин.

## Юрлица и группы

- Есть две группы юрлиц:
  - `BK_VM` — `ООО "Бургер БК"` и `ООО "Воглия Матта"`
  - `PS` — `ООО "Пицца Стар"`
- Эта логика важна для фильтрации данных, прав доступа, ресторанов, поставщиков и справочников.
- На фронтенде она описана в `src/lib/legalEntities.js`.
- На бэкенде она описана в `api/includes/legal_entities.php`.

Важно:

- Перед изменениями, связанными с юрлицами, проверять обе стороны.
- Нельзя менять такую логику только на фронтенде или только на бэкенде.

## Права доступа

- Права и роли размазаны по нескольким местам:
  - `src/stores/userStore.js`
  - `src/router/index.js`
  - `api/includes/helpers.php`
  - `api/includes/crud.php`
  - частично `api/includes/rpc.php`

Важно:

- Любые изменения в ролях, видимости модулей, доступах к страницам или таблицам проверять и на фронтенде, и на бэкенде.

## Сборка, миграции, тесты

- Автотестов нет.
- Линтера нет.
- Миграции применяются вручную.
- Отдельные разовые миграции могут быть SQL-файлами или PHP-скриптами.
- При изменении схемы БД обычно нужно добавлять новый файл в `migrations/`.

## Что не редактировать как исходники

- Не вносить рабочие изменения в `dist/`.
- Не редактировать собранные файлы из `public/assets/`.
- Не считать `public/index.html` исходной точкой приложения, если задача относится к фронтенду Vue.

Править нужно исходники в `src/`, `api/`, `migrations/`, `scripts/`.

## Зоны повышенного риска

- `api/includes/rpc.php` — очень большой файл, здесь легко сломать соседние сценарии.
- Логика юрлиц продублирована между фронтом и бэком.
- Права доступа и RBAC распределены по нескольким файлам.
- Из-за отсутствия тестов любые изменения требуют аккуратной ручной проверки.
- Миграции лежат в двух папках, поэтому легко создать рассинхрон.

## Практика внесения изменений

- Если задача относится к конкретному разделу интерфейса, обычно править соответствующий `src/views/*` и связанный `src/stores/*`.
- Если задача связана с расчётами заказа, начинать с `src/stores/orderStore.js`, `src/lib/calculations.js`, `src/lib/saveOrder.js`.
- Если задача связана с доступами, сессиями или ролями, проверять `userStore`, router guard и PHP RBAC.
- Если задача затрагивает ресторанный кабинет, заявки поставщикам или загрузку машин, почти всегда нужны правки и на фронтенде, и на соответствующем backend-модуле.
